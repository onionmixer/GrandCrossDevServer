# PLAN_05 — OS별 경로 처리 정책

경로 문제는 "번역 메커니즘"(§2)과 "각 OS의 제약"(§3)으로 나뉜다.
원칙부터 명확히 한다.

## 1. 원칙

1. **데몬은 경로를 해석하지 않는다.** CWD 인자도, 명령행 속 경로도
   원격 OS에 그대로 전달된다 (opaque). 모든 번역은 클라이언트에서.
2. **클라이언트가 번역하는 것은 CWD 하나뿐이다.** 명령행 인자 속
   경로는 번역하지 않는다 — 어디까지가 경로인지 신뢰성 있게 판별할
   수 없기 때문(마법적 치환은 오동작 시 추적이 더 어렵다).
3. 따라서 **상대경로 원칙**: 매핑된 CWD 아래에서 상대경로로 작업
   하는 것이 이식 가능한 사용법이다.
   ```sh
   cd /mnt/proj/src && gcds win "cl foo.c"      # O (상대경로)
   gcds win "cl /mnt/proj/src/foo.c"            # X (로컬 절대경로)
   gcds win "cl Z:\proj\src\foo.c"              # O (원격 네이티브 —
                                               #    사용자 책임)
   ```
   함정: MSVC `cl`은 `/`로 시작하는 인자를 옵션으로 해석한다.
   Windows 대상 상대경로는 `sub\foo.c` 또는 `.\sub\foo.c` 형태로.
4. 에러 메시지 속 원격 경로의 **역변환**(원격→로컬)은 클라이언트
   출력 필터(`--mapback`)로 제공한다 (선택 기능 — §4).

## 2. 매핑 규칙 (pathmap, 정밀 명세)

```
host.<alias>.map.<n> = <local-prefix>|<remote-prefix>[|<sep>]
```

- **컴포넌트 경계 매칭**: local-prefix는 경로 컴포넌트 경계에서만
  일치한다. `/mnt/proj` 매핑은 `/mnt/project`에 일치하지 않는다.
- **최장 일치 우선**: 여러 매핑이 일치하면 local-prefix가 가장 긴
  것을 쓴다 (`/mnt`와 `/mnt/proj`가 있으면 후자).
- local-prefix 끝의 `/`는 무시된다 (`/mnt/proj/` == `/mnt/proj`).
- **구분자 결정**: 3번째 필드 `<sep>`(한 글자)가 있으면 그것을,
  없으면 remote-prefix에 `\`가 포함될 때 `\`, 아니면 `/`.
  나머지 경로의 `/`가 이 구분자로 치환된다.
  - 고전 Mac(Phase 4)용: `map.1 = /mnt/mac|MacHD:proj|:`
- remote-prefix가 구분자로 끝나면 이중 구분자가 생기지 않도록
  접합부에서 하나로 줄인다.
- cwd가 local-prefix와 정확히 같으면 remote-prefix가 그대로 CWD.
- 어떤 매핑에도 일치하지 않으면 CWD를 보내지 않는다 (원격 데몬의
  현재 디렉토리에서 실행 — 데몬 기동 디렉토리 또는 직전 세션의
  CWD. 재현성이 필요하면 매핑을 정의할 것).

## 3. OS별 제약 (사용자가 알아야 할 것)

| 항목 | Linux | Windows | macOS | BeOS | NeXTSTEP | MS-DOS |
|------|-------|---------|-------|------|----------|--------|
| 대소문자 | 구분 | 비구분 | 기본 비구분 | 구분 | 구분 | 비구분 |
| 구분자 | / | \ (/ 일부 허용) | / | / | / | \ |
| 파일명 | 255B | 255자 | 255자 | 255B | 255B | **8.3** |
| 경로 길이 | ~4096 | 260 (MAX_PATH) | ~1024 | ~1024 | ~1024 | **~66** |

- **대소문자**: 소스 트리는 전부 소문자 파일명 + #include 대소문자
  일치를 권장. Linux에서 빌드되는 `Foo.h`/`foo.h` 혼용은
  비구분 FS(Windows/macOS/DOS)에서 충돌하거나, 반대로 비구분 FS
  에서만 빌드되는 코드가 Linux 대상에서 깨진다.
- **DOS 8.3**: DOS를 이식 대상에 포함하는 프로젝트는 소스/산출물
  파일명이 8.3을 지켜야 한다 (공유 스토리지 경유 시 긴 이름은
  뭉개져 보이거나 접근 불가). 본 프로젝트 소스가 8.3인 이유.
- **Windows UNC**: `\\server\share`는 cmd.exe의 작업 디렉토리가
  될 수 없다 (C:\로 떨어짐). **공유 폴더는 네트워크 드라이브
  문자로 마운트하고 드라이브 경로로 매핑할 것** (`Z:\proj`).
- Windows 드라이브 상대 경로(`Z:foo`)는 사용하지 않는다 (미정의).
- 심볼릭 링크: 클라이언트의 getcwd()는 해석된(실제) 경로를
  반환하므로, 매핑의 local-prefix도 실제 경로로 적을 것.

## 4. 역변환 필터 (--mapback)

- `gcds`가 O/E 스트림을 라인 단위로 스캔, remote-prefix로 시작하는
  토큰을 local-prefix로 치환하고 구분자를 되돌린다.
- 기본 꺼짐(`--mapback` 또는 conf `mapback=1`) — 바이너리 출력을
  깨뜨릴 수 있으므로 명시적으로 켠다. 켜져 있어도 O/E 중 텍스트로
  보이는 라인에만 적용.
- 대소문자 비구분 OS의 경로는 비구분 비교로 매칭한다.

## 5. 프로젝트 구성 지침 (요약)

- 매핑된 CWD + 상대경로로만 빌드 스크립트를 작성한다.
- 파일명: 소문자 통일, 공백 없음, 이식 하한이 DOS면 8.3.
- Makefile 안의 경로도 상대경로로 — 절대경로가 박히면 그 OS
  전용이 된다.
