# PLAN_01 — GCDSP 프로토콜 명세 (v1)

GCDSP = CrossDev Server Protocol. TCP 위의 평문 라인 기반 프로토콜.

## 1. 일반 규칙

- 전송 계층: **신뢰 가능한 양방향 바이트 스트림.** 두 가지 구현:
  - TCP, IPv4. 기본 포트 **9910** (설정으로 변경 가능).
  - 시리얼 RS-232, 기본 **9600 8N1** (§1.1의 특칙 적용).
  이하 명세는 두 채널에서 동일하다.
- 문자 집합: 제어 라인은 7bit ASCII. 페이로드(출력 데이터)는
  임의의 바이트열(8bit clean).
- **라인**: LF(0x0A)로 종료. 수신 측은 라인 내 CR(0x0D)을 무시한다.
  (Windows 쪽 구현/디버깅 편의)
- 라인 최대 길이: **1024바이트**(LF 포함). 초과 시 프로토콜 위반으로
  세션 종료. 모든 구현은 고정 버퍼를 쓸 수 있다.
- 토큰 구분: 단일 스페이스(0x20). 명령어는 대문자.
- 한 세션 = 한 TCP 연결. 세션 안에서 여러 `RUN`을 순차 실행할 수 있다.
- 알 수 없는 명령 수신 시: `ERR 500 unknown command` 응답 후 세션 유지.
  (전방 호환: 구버전 데몬에 신명령을 보내면 오류로 감지 가능)

### 1.1 시리얼 채널 특칙

TCP와 달리 시리얼에는 연결 수립/해제 개념이 없으므로 다음을 더한다.

- **세션 시작 = HELLO attention.** 데몬은 유휴 상태에서 라인을 읽으며
  대기하고, `HELLO` 라인을 수신하면 greeting(`GCDS 1 ...`)을 보내
  세션을 시작한다. `HELLO` 이외의 입력(라인 노이즈 포함)은
  조용히 버린다 — 이것이 재동기(resync) 규칙이다.
- **세션 종료 = QUIT 또는 프로토콜 위반.** 이후 데몬은 HELLO 대기로
  복귀한다. TCP의 "연결 끊김" 신호가 없으므로, 클라이언트가 죽으면
  데몬은 현재 세션에 머문다 — 클라이언트는 재기동 시 `HELLO`를
  다시 보내기 전에 `QUIT`을 먼저 보내 잔류 세션을 정리한다.
  (AUTH 전 상태의 QUIT은 항상 허용: `OK` 후 HELLO 대기 복귀)
- **프레임 크기 정책**: 시리얼에서 송신자는 O/E/I 프레임을
  **256바이트 이하**로 쪼갠다. (문법은 동일 — 4096은 TCP 상한,
  256은 시리얼 권장 상한. 9600bps에서 프레임당 ~0.27초)
- **흐름 제어**: 하드웨어(RTS/CTS)를 권장, 지원되지 않으면 없음.
  **XON/XOFF(소프트웨어)는 금지** — 바이너리 프레임 페이로드와
  충돌한다. 9600bps는 느려서 무흐름제어 + blocking 읽기로 충분하다.
- **오류 검출 없음 (v1)**: 내부 단거리 케이블 전제. 프레임 CRC는
  capability 확장(`CRC`)으로 예약만 해 둔다.
- **gcdslog와 포트 겸용 불가**: GCDSP 대화 채널과 커널 콘솔 raw 캡처
  (PLAN_04 §3)는 용도가 다르다. 같은 포트에 동시 사용할 수 없다.
- 비동기 모델(§7)과의 결합: DOS 시리얼 데몬도 동일하게 동작한다.
  "실행 중 connect 실패" 대신 "실행 중 HELLO 무응답"이 되고,
  클라이언트 폴링 규칙은 같다 (HELLO 재시도).

## 2. 응답 형식

- 성공: `OK` 또는 `OK <추가정보>`
- 실패: `ERR <3자리코드> <메시지>`
  - 4xx: 클라이언트 요청 문제 (인증 실패, 잘못된 인자 등)
  - 5xx: 서버측 문제 (실행 실패, 자원 부족 등)

## 3. 세션 흐름

```
연결 수립
S: GCDS 1 <os-tag> <hostname>          ; 인사. 프로토콜 버전 1
C: AUTH <token>
S: OK                                  ; 실패 시 ERR 401 후 즉시 연결 종료
C: CWD /boot/home/proj
S: OK                                  ; chdir 실패 시 ERR 404
C: ENV CC=gcc                          ; 0회 이상 반복
S: OK
C: RUN make -j1 all
S: O 512                               ; stdout 512바이트 프레임 (아래 §5)
S: <512 raw bytes>
S: E 87
S: <87 raw bytes>
S: ...                                 ; O/E 프레임 반복, 순서는 발생순
S: X 0                                 ; 종료 코드 0
C: RUN ./testsuite                     ; 같은 세션에서 추가 실행 가능
S: ...
S: X 1
C: QUIT
S: OK                                  ; 이후 양측 연결 종료
```

## 4. 명령 상세

### 서버 인사 (연결 직후 서버가 먼저 송신)
```
GCDS <proto-ver> <os-tag> <hostname> [<cap> ...]
```
- `proto-ver`: 정수. 본 명세는 `1`.
- `os-tag`: `win32` | `macos` | `beos` | `nextstep` | `dos` | `posix` 등
  빌드 시 결정되는 식별자. 클라이언트의 진단 출력용.
- `cap`: 0개 이상의 capability 플래그. 클라이언트는 모르는 플래그를
  무시한다. 본 명세가 정의하는 플래그:
  - `ASYNC` — 이 데몬은 비동기 작업 모델(§7)을 권장/요구한다.
    `gcds`는 이 플래그를 보면 RUN 대신 RUNA/RESULT 흐름을 자동 사용한다.
  - `INTERACTIVE` — 이 데몬은 대화형 실행(RUNI, §5.1)을 지원한다.
    소켓/자식 출력의 동시 감시가 가능한 플랫폼만 표기한다
    (POSIX/Win32/Haiku/NeXTSTEP. 고전 BeOS R5·DOS는 불가 — PLAN_02 §4).
  - `UTF8` — **텍스트가 UTF-8임을 보장**한다(로컬 인코딩이 다르면
    데몬이 변환). 텍스트란 O/E 프레임, 명령행, CWD/ENV 값, I 프레임을
    말하며 **D 프레임(PUT/GET)은 규약 밖의 raw 바이트**다. ASCII만
    가정하는 DOS/NeXTSTEP은 표기하지 않는다 (PLAN_02 §1.1).
  - `LIVE` — 작업 실행 중에도 제어 세션(§4.1)을 수락한다.
    멀티태스킹 OS(POSIX/Win32/BeOS·Haiku/NeXTSTEP)의 데몬이 표기.
    단일 태스킹(DOS)은 불가.

### AUTH — 인증 (세션당 1회, 최초 명령이어야 함)
```
AUTH <token>
```
- token: 스페이스를 포함하지 않는 ASCII 문자열, 최대 255바이트.
- 실패 시 `ERR 401 auth failed` 송신 후 서버가 연결을 닫는다.
- AUTH 이전에 다른 명령이 오면 `ERR 401` 후 연결 종료.

### CWD — 작업 디렉토리 지정
```
CWD <원격 네이티브 경로>
```
- 경로는 스페이스를 포함할 수 있다 (첫 스페이스 이후 라인 끝까지).
- 데몬이 즉시 `chdir()` (Win32: `SetCurrentDirectory`) 수행.
- 실패: `ERR 404 no such directory`

### ENV — 환경변수 설정
```
ENV <NAME>=<VALUE>
```
- 이후의 RUN에 적용된다. 같은 NAME 재설정 시 덮어쓴다.
- 세션당 최대 32개. 초과: `ERR 431 too many env`

### RUN — 명령 실행
```
RUN <명령행>
```
- 첫 스페이스 이후 라인 끝까지가 명령행. 데몬은 이를 OS 기본 셸에
  전달한다: POSIX `/bin/sh -c "<명령행>"`,
  Win32 `cmd.exe /c <명령행>` (설정 `shell=`로 교체 가능).
- 응답은 §5의 출력 프레임 시퀀스이며 반드시 `X` 프레임으로 끝난다.
- 실행 자체가 불가능한 경우(셸 없음 등): `ERR 500 exec failed`
  (이 경우 X 프레임 없음)
- **클라이언트는 RUN 송신 후 X 프레임 수신까지 아무것도 보내지
  않아야 하며, 이 구간에 수신 타임아웃을 걸어서는 안 된다.**
  (단일 태스킹/폴링 스택 플랫폼에서 실행 중 데몬이 네트워크에
  응답할 수 없기 때문. 모든 플랫폼 공통 규정으로 둔다.
  실행 중 입력/중단이 필요하면 RUN이 아니라 RUNI를 쓴다 — §5.1)
- RUN 처리 중 클라이언트 연결이 끊기면 데몬은 가능한 수단으로
  자식 프로세스를 정리하고 다음 accept로 돌아간다.

### PING — 생존 확인 (alive check)
```
PING        →  OK
```
- **`LIVE` 데몬은 작업 실행 중에도 제어 세션(§4.1)으로 응답한다.**
  따라서 LIVE 호스트의 무응답은 "다운"으로 해석할 수 있다.
- LIVE가 없는 데몬(단일 태스킹 — DOS)은 유휴 상태에서만 응답
  가능하며, 무응답은 busy와 dead를 구분하지 않는다 (PLAN_00 D11).
- 클라이언트는 `gcds --ping <host>`로 노출하고, 무응답 시 호스트의
  LIVE 여부에 따라 "다운" 또는 "busy 또는 다운"으로 보고한다.

### STAT — 상태 확인 (status check)
```
STAT
→ OK idle                  ; 유휴, 보관 결과 없음
→ OK busy <jobid>          ; 작업 실행 중 (LIVE 데몬의 제어 세션 응답.
                           ;  동기 RUN/RUNI 실행 중이면 jobid는 0)
→ OK result <jobid>        ; 회수 대기 중인 비동기 결과 보유 (§7)
```
- AUTH 이후에만 허용 (401 규칙 동일).
- `OK busy`는 LIVE 데몬에서만 관찰된다. 단일 태스킹 데몬은 실행 중
  응답 자체가 불가능하므로 busy가 무응답으로 나타난다.
- 데몬 재시작 감지 용도: 클라이언트가 기대한 jobid가 `OK idle`로
  돌아오면 결과가 소실된 것이다 (RESULT의 ERR 404와 동등 정보를
  결과 회수 전에 알 수 있음).

### 4.1 제어 세션 (LIVE capability)

멀티태스킹 OS의 데몬은 작업(RUN/RUNI/RUNA 처리) 실행 중에도 새
연결을 수락해야 한다 — 상태와 생존이 실시간으로 보여야 하기 때문.
이때 수락되는 연결이 **제어 세션**이다.

- 제어 세션에서 허용되는 명령: `AUTH`, `PING`, `STAT`, `QUIT`.
  그 외(RUN/RUNA/RUNI/CWD/ENV/RESULT)는 `ERR 409 busy`로 거부한다.
  (RESULT까지 거부하는 이유: 실행 중인 작업이 결과 임시 파일을
  덮어쓰고 있을 수 있음)
- 제어 세션은 한 번에 하나씩 순차 처리한다(반복 서버 원칙 유지).
  대기 중 추가 연결은 listen backlog에 머문다.
- 구현: 실행 감시 루프(select/폴링)에 listen 소켓을 포함
  (PLAN_02 §5.2). LIVE는 자식을 비동기로 감시하는 실행 구조
  (live.c / live_w32.c)에서 표기한다.
- 단일 태스킹 플랫폼(DOS)은 구조적으로 LIVE 불가 — 실행 중 연결
  시도는 무응답이고, busy/dead 구분은 실행 종료 후 폴링으로.
- **LIVE는 TCP 모드 한정**이다. 시리얼 모드(§1.1)는 채널이 하나뿐
  이라 실행 중 제어 세션이 들어올 통로가 없다 — 멀티태스킹 OS라도
  시리얼 데몬은 LIVE를 표기하지 않는다.

### QUIT — 정상 종료
```
QUIT        →  OK  (이후 연결 종료)
```

### RUNI — 대화형 실행 (INTERACTIVE capability 필요, §5.1)
```
RUNI <명령행>
```
- 실행 방식은 RUN과 같으나, 실행 중 클라이언트가 `I`(stdin)/`K`(중단)
  프레임을 보낼 수 있고 데몬이 이를 자식에 전달한다.
- 출력은 RUN과 동일한 O/E/X 프레임. 단, 자식 종료 대기 없이
  발생 즉시 스트리밍된다 (대화형의 전제 조건).
- `INTERACTIVE` capability가 없는 데몬은 `ERR 501 not supported`로
  거부한다. 클라이언트는 greeting에서 미리 판별할 수 있다.
- 자식의 stdin/stdout/stderr는 **파이프**다 (pty 아님). tty 여부로
  동작이 달라지는 도구의 한계와 우회는 PLAN_04 §5.

### RUNA — 비동기 실행 접수 (§7)
```
RUNA <명령행>
```
- 의미는 RUN과 같으나, 데몬은 실행하지 않은 채
  `OK <jobid>`를 응답하고 **연결을 닫은 뒤** 실행을 시작한다.
- `jobid`: 1..32767 순환 증가 정수 (메모리 유지, 데몬 재시작 시 소실).
- 접수 시점의 CWD/ENV(같은 세션에서 먼저 보낸 것)가 작업에 적용된다.
- 이미 미회수 결과가 있어도 새 RUNA는 그것을 덮어쓴다
  (단일 작업 슬롯 — 결과 보존이 필요하면 먼저 RESULT로 회수할 것).

### RESULT — 비동기 결과 회수 (§7)
```
RESULT <jobid>
```
- 성공: §5와 동일한 O/E/X 프레임 시퀀스.
- `ERR 404 no such result` — 모르는 jobid이거나(데몬 재시작 포함)
  보관된 결과가 없음.
- 결과는 다음 RUNA/RUN 접수 전까지 보관되며, RESULT는 **멱등**이다
  (재회수 가능 — 회수 도중 연결이 끊겨도 재시도로 복구).

### PUT — 파일 업로드 (클라이언트 → 서버)
```
PUT <원격경로>
S: OK                        ; 쓰기 준비됨 (파일 생성 성공)
   또는 ERR 4xx              ; 경로 열기 실패 등 (이 경우 전송 없음)
C: D <len>\n<len바이트 raw>  ; 데이터 프레임, 반복 (1 <= len <= 4096)
C: D 0\n                     ; EOF 표시 (전송 종료)
S: OK <total>                ; total 바이트 기록 완료
   또는 ERR 5xx              ; 쓰기 오류 (부분 파일이 남을 수 있음)
```
- 경로는 CWD와 동일하게 **불투명**하다 (원격 OS 네이티브 경로,
  현재 작업 디렉토리 기준 상대경로 가능). 데몬은 그대로 fopen("wb").
- `maxout`이 설정돼 있으면 그 값을 PUT 총량 상한으로도 적용
  (초과 시 `ERR 413`, 부분 파일 삭제).
- 바이너리 안전: D 프레임은 O/E와 동일한 8bit clean 길이접두.
  시리얼에서는 프레임을 256B 이하로 쪼갠다(§1.1).

### GET — 파일 다운로드 (서버 → 클라이언트)
```
GET <원격경로>
S: OK                        ; 파일 열림
   또는 ERR 404              ; 없는 파일 등
S: D <len>\n<len바이트 raw>  ; 데이터 프레임, 반복
S: D 0\n                     ; EOF
```
- 경로 규칙은 PUT과 동일. 데몬은 fopen("rb").
- 용도: 공유 파일시스템이 없는 원격지(DOS 등)에서 소스 업로드 →
  원격 컴파일(RUN) → 산출물 다운로드. 클라이언트 `--put`/`--get`.
- PUT/GET은 임의 파일 쓰기/읽기이므로 RUN과 동일한 신뢰 수준
  (토큰 + `allow`)을 전제로 한다.

## 5. 출력 프레임 (RUN 응답)

바이너리 안전을 위해 길이 접두(length-prefixed) 프레임을 쓴다.

```
O <len>\n<len바이트 raw>     ; stdout 데이터, 1 <= len <= 4096
E <len>\n<len바이트 raw>     ; stderr 데이터, 1 <= len <= 4096
X <exitcode>\n               ; 실행 종료. exitcode는 0..255 정수
```

- raw 바이트 뒤에는 구분자가 없다 (다음 프레임 헤더가 바로 이어짐).
- `len` 상한 4096: 수신측이 고정 버퍼로 처리 가능하게 하는 제한.
- 블로킹 캡처 구현은 명령 종료 후 out.tmp 전체를 O 프레임 연속으로,
  err.tmp 전체를 E 프레임 연속으로 보낸 뒤 X를 보낸다. 스트리밍
  구현은 발생 순서대로 교차 송신한다.
  **클라이언트는 두 경우를 구분할 필요가 없다** (같은 문법).
- exitcode 규약: POSIX는 `WEXITSTATUS`, 시그널 종료는 `128+signo`.
  Win32는 프로세스 종료 코드를 255로 클램프.

### 5.1 클라이언트 프레임 (RUNI 실행 중 전용)

RUNI의 X 프레임 수신 전까지, 클라이언트는 아래 프레임을 보낼 수 있다.
(RUN/RUNA에서는 금지 — 침묵 규정 적용)

```
I <len>\n<len바이트 raw>     ; 자식 stdin으로 전달, 1 <= len <= 4096
I 0\n                        ; stdin EOF — 데몬이 자식 stdin을 닫음
K\n                          ; 자식 중단 요청
```

- `K` 수신 시 데몬은 자식을 강제 종료한다
  (POSIX: SIGTERM → 유예 후 SIGKILL / Win32: TerminateProcess).
  이후 남은 출력을 flush하고 X 프레임으로 세션을 정상 마감한다.
- K로 종료된 자식의 X 값은 플랫폼 규약을 따른다(POSIX 128+signo,
  Win32 255). 클라이언트는 자신이 K를 보냈다는 사실로 "사용자 중단"을
  판별하며 X 값에 의존하지 않는다.
- `K` 이후 도착하는 `I` 프레임은 무시된다.
- 데몬은 RUNI 동안 소켓 수신과 자식 출력 수집을 동시에 처리해야
  한다. 구현 방식은 플랫폼별로 다르며(PLAN_02 §5.1) 프로토콜은
  규정하지 않는다.

## 6. 타임아웃 (보수적 규정)

- 구세대 OS에서 `SO_RCVTIMEO`/`select()` 동작을 신뢰할 수 없으므로,
  **v1 데몬의 필수 요구는 blocking I/O만으로 동작하는 것**이다.
- 데몬: AUTH 대기 중 무한 대기를 피하고 싶은 플랫폼에서는
  POSIX `alarm()`을 선택적으로 사용(기본 30초). 불가한 플랫폼은
  무기한 대기 허용 (내부망 전제로 수용).
- 클라이언트(Linux 한정이므로 자유로움): connect/응답 대기에
  `select()` 기반 타임아웃 사용 가능.

## 7. 비동기 작업 모델 (ASYNC capability, MS-DOS 등)

단일 태스킹 플랫폼(대표: MS-DOS + Watt-32)에서는 자식 실행 중 데몬이
TCP 스택을 tick할 수 없으므로, 접수와 회수를 별도 세션으로 분리한다.

```
[세션 1 — 접수]
S: GCDS 1 dos DOSBOX1 ASYNC
C: AUTH <token>          → OK
C: CWD C:\PROJ           → OK
C: ENV ...               → OK
C: RUNA wcl -ml prog.c
S: OK 17                  ; jobid=17 접수
S: (연결 종료)             ; 데몬이 닫고 즉시 실행 시작

[실행 중]
- 데몬은 listen하지 않는 것과 같다. 이 동안의 connect 시도는
  무시되거나(폴링 스택: SYN 무응답 → 타임아웃) 지연 후 수락된다
  (멀티태스킹 OS에서 같은 모델을 구현한 경우: backlog 대기).
  클라이언트는 두 경우 모두 재시도로 처리한다.

[세션 2..N — 폴링/회수]
C: (connect 재시도: 시도당 타임아웃 ~10초, 간격 2~5초, 총 한도는
    클라이언트 설정. 연결 실패는 오류가 아니라 "아직 실행 중"이다)
S: GCDS 1 dos DOSBOX1 ASYNC
C: AUTH <token>          → OK
C: STAT                  → OK result 17   ; (선택) 회수 전 확인 —
                                          ;  OK idle이면 데몬 재시작으로
                                          ;  결과 소실로 판정
C: RESULT 17
S: O/E/X 프레임           ; §5와 동일 — 이후 QUIT
```

- `gcds`는 ASYNC 플래그를 보면 이 흐름을 자동 수행하고, 최종적으로
  동기 RUN과 동일하게 stdout/stderr/exit code를 로컬에 반영한다.
- ASYNC 데몬에서도 RUN(동기)은 유효하다 — 실행이 짧아 연결 유지가
  문제없는 명령용. 어느 쪽을 쓸지는 전적으로 클라이언트 판단이며,
  기본값은 "ASYNC 플래그가 있으면 RUNA".
- RESULT 응답의 X 프레임까지 수신 완료한 뒤에도 결과는 다음 작업
  접수 전까지 서버에 남는다(멱등 재회수 — §4 RESULT 참조).

## 8. 프로토콜 위반 처리

- 라인 길이 초과, AUTH 전 명령, 잘못된 프레임 — 데몬은
  `ERR 400 protocol violation`을 보낼 수 있으면 보내고 연결을 닫은 뒤
  accept 루프로 복귀한다. 데몬 프로세스는 어떤 입력에도 죽지 않는다.
